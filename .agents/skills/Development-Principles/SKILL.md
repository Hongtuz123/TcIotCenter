\# 微感測器 GIS Dashboard 規劃文件



\## 1. 專案名稱

\*\*微感 GIS Dashboard\*\*

以環境部環境物聯網空氣品質微感測器資料為核心，建立一個以地圖為主軸的全端網頁平台，用於觀察空氣品質異常、快速定位疑似燃燒事件，並支援後續 API 自動更新與事件管理。



\---



\## 2. 專案目標



本專案目標是將環境部環境物聯網的空氣品質微感測器資料，轉換為可視化、可查詢、可追蹤的 GIS dashboard。  

使用者可以透過地圖快速查看空間分布，找出可能存在燃燒或異常排放的區域，並搭配事件清單進行人工標註與後續追蹤。  

初期先匯入 1 個月歷史資料，完成系統架構與視覺化介面後，再串接即時 API 做自動更新。  

前端採全端網頁架構，並規劃後續部署到 GitHub 與 Vercel，作為可展示的 demo 頁面。\[web:1]\[web:4]\[web:3]



\---



\## 3. 使用情境



1\. 使用者打開系統後，先看到台灣地圖與微感測器點位分布。

2\. 使用者可點選任一感測點，查看該點的時間序列、測項與測值。

3\. 系統可依空間與時間條件篩選出異常值，協助判斷疑似燃燒熱點。

4\. 使用者可將疑似事件建立為「事件」並加入事件清單，進行後續追蹤。

5\. 管理者可在後續串接 API 後，自動更新資料並持續維運。\[web:4]\[web:1]



\---



\## 4. 資料欄位設計



你預計的核心欄位如下：



\- 座標：經緯度，作為地圖點位顯示與空間查詢依據。

\- 時間：資料時間戳記，用於趨勢分析與事件比對。

\- 測項：例如 PM2.5、PM10、溫度、濕度或其他微感測器項目。

\- 測值：對應測項的數值。



建議在資料表中再補充以下欄位，以提升後續分析能力：



\- sensor\_id：感測器唯一識別碼。

\- location\_name：點位名稱或行政區。

\- source：資料來源，方便區分歷史匯入與 API 即時資料。

\- status：資料狀態，例如正常、疑似異常、待確認。

\- event\_id：關聯事件編號，若該筆資料已納入事件分析。\[web:1]\[web:4]

\## 5. 系統架構



\### 5.1 前端

建議採用單頁式或模組化的全端網頁架構，前端負責：



\- 地圖視覺化。

\- 感測器資料篩選。

\- 異常熱點檢視。

\- 事件清單管理。

\- 圖表與明細面板顯示。



\### 5.2 地圖引擎

地圖 API 指定使用 \*\*Mapbox\*\*，用來呈現：



\- 感測器點位。

\- 熱點聚合。

\- 異常區域圈選。

\- 使用者拖曳、縮放、定位與圖層切換。



Mapbox GL JS 可作為網頁地圖的主要實作方案。\[web:2]\[web:5]



\### 5.3 後端

後端建議提供：



\- 資料匯入 API。

\- 事件建立與更新 API。

\- 感測資料查詢 API。

\- 空間篩選 API。

\- 異常判定 API。



若未來要接環境部環境物聯網資料，可先用批次匯入 1 個月歷史資料，再切換成定期同步或排程更新。\[web:1]\[web:4]



\### 5.4 資料庫

建議使用關聯式資料庫來儲存：



\- 感測器主檔。

\- 原始觀測資料。

\- 事件主檔。

\- 事件與觀測資料的關聯表。

\- 使用者操作紀錄。

\## 7. 功能模組



\### 7.1 地圖主畫面

\- 顯示全台感測器點位。

\- 可依行政區、時間、測項、數值區間篩選。

\- 支援點擊查看單一感測器細節。

\- 支援地圖縮放後自動更新顯示密度。



\### 7.2 異常分析

\- 顯示高於門檻值的點位。

\- 支援時間窗分析，例如近 1 小時、近 6 小時、近 24 小時。

\- 顯示疑似燃燒熱點聚集區。

\- 可用顏色或圖示標示風險等級。



\### 7.3 事件清單

\- 新增事件。

\- 編輯事件內容。

\- 指派事件狀態，例如待確認、調查中、已結案。

\- 事件可綁定多筆觀測資料與地圖區域。

\- 事件清單可依時間排序與狀態篩選。



\### 7.4 圖表面板

\- 顯示趨勢折線圖。

\- 顯示單點多測項比較。

\- 顯示事件前後變化。

\- 顯示異常值次數統計。



\### 7.5 資料匯入與更新

\- 初期匯入 1 個月歷史資料。

\- 後續串接 API，自動更新新資料。

\- 支援手動重新匯入與資料校正。

\- 支援重複資料去重與資料品質檢查。\[web:1]\[web:4]

\## 8. GIS 設計原則



因為你的內容核心是「透過空間找出燃燒事件」，所以介面設計建議以 GIS 為主，而不是一般表格型 dashboard。  

地圖應放在畫面中央，左側放條件篩選，右側放事件清單與詳細資訊。  

點位顏色建議依測值區間分級，例如低、正常、偏高、高、極高。  

若同一區域在短時間內出現連續異常，可自動形成熱區，方便使用者快速判讀。\[web:5]\[web:2]

\## 12. 異常判斷邏輯



建議先以規則式方式建立初版判斷：



\- 單點值超過門檻。

\- 同區域多點同步偏高。

\- 同一感測器在短時間內快速上升。

\- 風向或空間鄰近點出現連鎖異常。



這些規則可以先作為「疑似燃燒」提示，讓使用者透過地圖快速查找，再進一步人工確認。  

後續若資料量足夠，可再加入統計模型或機器學習辨識熱點模式。

\## 13. Antigravity 建置建議



建議在 Antigravity 中直接建立以下結構：



\- `app/`：前端頁面。

\- `components/`：地圖、事件清單、圖表元件。

\- `lib/`：資料處理與 API 工具。

\- `pages/` 或 `app/` 路由：地圖頁、事件頁、分析頁、設定頁。

\- `api/`：後端 API 路由。

\- `styles/`：全域樣式。

\- `data/`：初期匯入的歷史資料樣本。



若要快速產出 demo，建議先完成：

1\. 地圖主頁。

2\. 事件清單頁。

3\. 假資料匯入。

4\. Mapbox 串接。

5\. Vercel 部署設定。\[web:2]\[web:3]
